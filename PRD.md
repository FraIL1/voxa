# «Voxa» — платформа голосового и текстового общения для закрытого круга друзей

Документ требований к продукту (PRD), техническая архитектура и план реализации.
Версия 1.1 — июль 2026. Автор: Артём.

## 1. Назначение продукта

Voxa (рабочее название) — самостоятельное приложение для голосового и текстового общения в закрытом сообществе друзей. Функционально это аналог Discord, но принципиально иной по масштабу и философии: продукт создаётся не для миллионов пользователей, а для одного приватного сообщества из 20–30 человек. Приложение разворачивается на арендованном сервере (VPS — виртуальный сервер у хостинг-провайдера, ~10–20 $/мес), который полностью подконтролен владельцу.

Ключевая идея: «свой уголок в интернете». Пользователи заходят в приложение, чтобы посидеть в голосовом канале с друзьями, переписываться, делиться файлами, мемами и ссылками, играть вместе и просто проводить время. Никакой рекламы, аналитики, продажи данных и внешней модерации — все данные лежат на сервере, доступ к которому есть только у владельца.

**Что продукт НЕ является:** это не публичная соцсеть, не мессенджер для незнакомцев и не коммерческий сервис. Регистрация только по приглашению. Масштабирование до тысяч пользователей сознательно не является целью первой версии — это упрощает архитектуру и удешевляет аренду сервера.

## 2. Цели

### 2.1 Цели продукта

- Дать группе из 20–30 друзей стабильную площадку для голосового общения с задержкой не хуже, чем в Discord (< 150 мс).
- Полный контроль над данными: всё хранится на арендованном VPS владельца, ничего не уходит третьим лицам.
- Работа в браузере и как десктоп-приложение (Windows/macOS/Linux).
- Простота эксплуатации: один человек (владелец) может развернуть и поддерживать систему, обновление — одной командой.
- Стоимость содержания — в пределах 10–20 $/мес за аренду VPS.

### 2.2 Измеримые критерии успеха

- В голосовом канале одновременно комфортно сидят 10+ человек без деградации звука.
- Доставка текстового сообщения всем онлайн-участникам — менее чем за 500 мс.
- Аптайм сервиса — 99% и выше (простой не более ~7 часов в месяц).
- Друзья реально пользуются: 10+ активных пользователей в неделю через месяц после запуска.

## 3. Целевая аудитория

Первичная аудитория — закрытый круг друзей владельца: 20–30 человек, возраст 15–35 лет, опытные пользователи Discord/Telegram. Сценарии: совместные игры, «посидеть на фоне» вечером, обсуждения, обмен мемами и файлами.

Из этого следуют требования: интерфейс должен быть привычен пользователю Discord (порог входа — нулевой), онбординг — по инвайт-ссылке за 1 минуту, никакой верификации телефона и прочей бюрократии.

Вторичная аудитория (перспектива): другие небольшие сообщества, которые захотят развернуть свой экземпляр, если проект будет опубликован как open-source.

## 4. Ключевые пользовательские сценарии

- **«Вечерний сбор».** Пользователь открывает приложение, видит, кто сидит в голосовом канале «Общий», заходит туда в один клик и общается. Остальные видят уведомление, что он подключился.
- **«Игровая сессия».** Компания сидит в канале «Игры», один включает демонстрацию экрана, остальные смотрят. Работают индикаторы «кто говорит», mute/deafen.
- **«Переписка днём».** Пользователи в течение дня пишут в текстовые каналы, кидают картинки и ссылки, ставят реакции, отвечают на сообщения. Вечером легко догнать пропущенное — есть маркер непрочитанного.
- **«Личный разговор».** Два пользователя переписываются в личных сообщениях и могут созвониться один на один.
- **«Приглашение новичка».** Владелец генерирует инвайт-ссылку, отправляет другу; тот переходит, регистрируется и сразу попадает в сообщество.

## 5. Функциональные требования (спецификация функций)

Приоритеты: **MVP** — первая рабочая версия; **v1.1** — сразу после запуска; **v2** — по желанию, когда основное стабильно.

### 5.1 Регистрация и аутентификация

**Назначение.** Контролируемый вход в закрытое сообщество. Регистрация возможна только по инвайт-ссылке.

**Поведение и требования:**

- Регистрация: инвайт-код + имя пользователя (уникальное, 3–24 символа) + пароль (минимум 10 символов, проверка по списку утёкших паролей).
- Вход по имени/паролю; сессия — пара JWT-токенов: access (15 мин) + refresh (30 дней, httpOnly cookie, ротация при каждом обновлении).
- Инвайт-ссылки создаёт владелец/модератор: с лимитом использований и сроком действия; список активных инвайтов с возможностью отозвать.
- Выход со всех устройств; смена пароля с завершением всех прочих сессий.
- Двухфакторная аутентификация (TOTP, приложение-аутентификатор) — опционально для пользователя, обязательно для владельца.
- Ограничение попыток входа: после 5 неудач — задержка с экспоненциальным ростом; лог попыток входа.

**Приоритет:** MVP (2FA — v1.1)

### 5.2 Сообщество, категории и каналы

**Назначение.** Структура пространства: одно сообщество («сервер»), внутри — категории, внутри категорий — текстовые и голосовые каналы.

**Поведение и требования:**

- В первой версии — ровно одно сообщество (мульти-серверность не нужна и сознательно исключена для простоты).
- Каналы двух типов: текстовые (#) и голосовые. Создание, переименование, удаление, перетаскивание между категориями — для ролей с правом «Управление каналами».
- У канала есть тема/описание; канал можно сделать приватным (виден только выбранным ролям).
- Стартовый набор при развёртывании: категория «Текст» (#общий, #мемы, #важное), категория «Голос» («Общий», «Игры», «AFK»).

**Приоритет:** MVP

### 5.3 Текстовый чат

**Назначение.** Основной канал асинхронного общения — обмен сообщениями в каналах в реальном времени.

**Поведение и требования:**

- Отправка/получение сообщений в реальном времени через WebSocket; мгновенное отображение у всех онлайн-участников.
- Markdown-форматирование: жирный, курсив, зачёркнутый, `код`, блоки кода с подсветкой, цитаты, спойлеры.
- Редактирование (с пометкой «изменено») и удаление своих сообщений; модераторы могут удалять чужие.
- Ответы (reply) с цитированием и переходом к оригиналу; упоминания @имя и @everyone с подсветкой.
- Реакции-эмодзи на сообщения; стандартный набор эмодзи + свои кастомные эмодзи сообщества (загрузка PNG до 256 КБ).
- История: бесконечная прокрутка с ленивой подгрузкой по 50 сообщений; маркер «непрочитанное с этого места».
- Индикатор «X печатает…»; предпросмотр ссылок (заголовок, описание, картинка); поиск по сообщениям канала.
- Закреплённые сообщения (список пинов канала).

**Приоритет:** MVP (поиск, пины, кастомные эмодзи — v1.1)

### 5.4 Голосовые каналы

**Назначение.** Сердце продукта. Постоянные голосовые комнаты, в которые можно зайти и выйти в любой момент.

**Поведение и требования:**

- Подключение к каналу в один клик; список участников канала виден всем ещё до входа.
- Аудио через WebRTC с SFU-сервером (LiveKit): каждый шлёт один поток вверх, сервер раздаёт остальным. Комфортно до 15–20 человек в одной комнате.
- Кодек Opus 48 кГц; целевая задержка < 150 мс; адаптация битрейта к качеству сети.
- Обработка звука: эхоподавление, шумоподавление, автоматическая регулировка усиления (встроенные в WebRTC + опционально RNNoise).
- Индикация говорящего (подсветка аватара); mute (микрофон) и deafen (полное отключение звука) в один клик.
- Индивидуальная громкость каждого участника (локально); выбор устройств ввода/вывода; режим Push-to-Talk с настраиваемой клавишей.
- Модерация: заглушить участника для всех, перенести в другой канал, отключить от голоса.
- Автопереброс в AFK-канал после N минут молчания (настраивается).

**Приоритет:** MVP (PTT, RNNoise, AFK — v1.1)

### 5.5 Демонстрация экрана и видео

**Назначение.** Совместный просмотр: показать игру, фильм или окно приложения; видео с камеры.

**Поведение и требования:**

- Демонстрация экрана/окна из голосового канала: 720p/30fps по умолчанию, до 1080p/60fps как настройка.
- Звук приложения при демонстрации (там, где платформа позволяет — Chrome/десктоп-клиент).
- Несколько одновременных демонстраций в канале; зритель выбирает, чей поток смотреть, окно можно развернуть.
- Видео с веб-камеры в голосовом канале (сетка участников).

**Приоритет:** v1.1 (камера — v2)

### 5.6 Личные сообщения

**Назначение.** Приватная переписка и звонки один на один вне общих каналов.

**Поведение и требования:**

- Диалог 1-на-1 с любым участником сообщества; те же возможности, что в каналах (markdown, файлы, реакции, ответы).
- Голосовой звонок 1-на-1 из диалога с входящим уведомлением-звонком.
- Групповые личные чаты до 10 человек.

**Приоритет:** v1.1 (групповые — v2)

### 5.7 Роли и права доступа

**Назначение.** Управление тем, кто и что может делать в сообществе.

**Поведение и требования:**

- Роли с цветом и порядком старшинства; у пользователя может быть несколько ролей, права суммируются.
- Набор прав: управление каналами, управление ролями, удаление чужих сообщений, кик/бан, заглушение в голосе, создание инвайтов, упоминание @everyone, загрузка файлов.
- Права можно переопределять на уровне конкретного канала (например, писать в #важное может только роль «Админ»).
- Стартовые роли: Владелец (все права, не удаляется), Модератор, Участник (по умолчанию).
- Цвет роли красит имя пользователя в чате и в списке участников.

**Приоритет:** MVP — базовые роли; переопределения по каналам — v1.1

### 5.8 Присутствие, статусы и уведомления

**Назначение.** Ощущение «живого» пространства: видно, кто онлайн и чем занят; важное не теряется.

**Поведение и требования:**

- Статусы: онлайн / неактивен (авто через 10 мин) / не беспокоить / невидимка; кастомный текстовый статус.
- Список участников с группировкой по ролям и статусам; у сидящих в голосе видно, в каком канале.
- Бэйджи непрочитанных на каналах; отдельный счётчик упоминаний.
- Десктоп-уведомления: упоминания и личные сообщения — всегда; все сообщения канала — по подписке на канал; «Не беспокоить» глушит всё.
- Звуковые сигналы: вход/выход участника в голосовой канал, входящий звонок, упоминание.

**Приоритет:** MVP (кастомный статус — v1.1)

### 5.9 Файлы и медиа

**Назначение.** Обмен изображениями, видео и документами.

**Поведение и требования:**

- Загрузка перетаскиванием, из буфера обмена (Ctrl+V) и через кнопку; лимит 50 МБ на файл (настраивается).
- Изображения и видео показываются инлайн с миниатюрами; клик — полноэкранный просмотр; остальные файлы — карточкой со значком, именем и размером.
- Хранение в S3-совместимом хранилище (MinIO на том же сервере); миниатюры генерируются на сервере.
- Проверка типа файла по содержимому (не по расширению); запрет исполняемых файлов; квота на пользователя.

**Приоритет:** MVP

### 5.10 Модерация и администрирование

**Назначение.** Инструменты владельца для поддержания порядка и контроля системы.

**Поведение и требования:**

- Кик и бан (по аккаунту) с указанием причины; список банов с разбаном.
- Таймаут («мут») пользователя на время: не может писать и говорить.
- Журнал аудита: кто удалил сообщение, создал канал, выдал роль, забанил и т.д.
- Панель владельца: список пользователей и сессий, активные инвайты, занятое место на диске, версия сервера.

**Приоритет:** MVP — кик/бан; остальное — v1.1

## 6. Фишки (отличительные особенности)

- **Полный суверенитет.** Арендованный сервер под полным контролем владельца: свои данные, свои правила. Никто не забанит сообщество и не изменит условия использования.
- **Настраиваемые звуковые панели (soundboard).** Каждый может загрузить короткие звуки-мемы и проигрывать их в голосовом канале горячей клавишей. (v1.1)
- **Комната «Кинотеатр».** Синхронизированный совместный просмотр: один стримит экран, у остальных — общий чат поверх, как в кино. (v2)
- **Статистика сообщества.** Кто сколько часов просидел в голосе, топ флудеров, график активности по дням. Только для своих, в шуточном тоне. (v2)
- **Свои эмодзи и звуки без лимитов «нитро».** Все возможности бесплатны для всех участников — монетизации нет по определению.
- **Лёгкость.** Цель — < 300 МБ ОЗУ у десктоп-клиента (Tauri, а не Electron).

## 7. Дизайн (UI/UX)

### 7.1 Принципы

- Привычность: раскладка знакома пользователю Discord — слева каналы, в центре чат, справа участники. Ноль обучения.
- Тёмная тема по умолчанию (плюс светлая на выбор); аккуратная, спокойная палитра.
- Скорость восприятия: главное действие — зайти в голос — доступно в один клик из любого места.
- Отзывчивость: оптимистичный UI (сообщение появляется мгновенно, до подтверждения сервера).

### 7.2 Макет главного экрана

Три колонки. Левая (240 px): название сообщества, дерево категорий и каналов, внизу — карточка пользователя с кнопками микрофона/наушников/настроек и, при подключении к голосу, панель текущего канала. Центральная: шапка канала, лента сообщений, поле ввода с кнопкой вложений. Правая (200 px, скрываемая): участники по ролям и статусам. В голосовом канале центральная область показывает сетку участников и демонстрации экрана.

### 7.3 Визуальный стиль

Собственная палитра, НЕ копирующая Discord (см. раздел 11). Тёмная тема: фон #1A1D24, панели #22262E, акцент — тёплый оранжевый #FF7A45 (у Discord — сиреневый #5865F2, сознательно уходим в другую гамму). Шрифт — Inter (SIL OFL). Иконки — Lucide (ISC). Скругления 8 px, лаконичные анимации 150 мс.

### 7.4 Адаптивность и доступность

Веб-версия адаптируется до ширины планшета (правая колонка скрывается, левая — в выдвижное меню). Полноценный мобильный клиент — вне рамок v1, но мобильный браузер должен позволять читать чат и заходить в голос. Горячие клавиши: mute (Ctrl+Shift+M), deafen (Ctrl+Shift+D), переключение каналов (Alt+↑/↓). Контраст текста — по WCAG AA.

## 8. Техническая архитектура

### 8.1 Выбранный стек

| Слой | Технология | Обоснование |
|---|---|---|
| Фронтенд (веб) | React 19 + TypeScript + Vite, Zustand, TanStack Query | Зрелая экосистема, быстрые сборки; один язык со всем проектом |
| Десктоп-клиент | Tauri 2 (обёртка над веб-клиентом) | В разы легче Electron; глобальные хоткеи, автозапуск, трей |
| Бэкенд (API + realtime) | Node.js 22 + NestJS + WebSocket (socket.io) | TypeScript на сервере, модульная структура |
| Голос/видео (SFU) | LiveKit (open-source, Apache 2.0) | Готовый продакшн-SFU на WebRTC; не пишем медиасервер сами |
| База данных | PostgreSQL 17 | Надёжность, полнотекстовый поиск |
| Кэш и pub/sub | Redis 7 | Присутствие, rate limiting, события |
| Файлы | MinIO (S3-совместимое) | Своё объектное хранилище на том же VPS |
| Обратный прокси | Caddy 2 | Автоматический HTTPS (Let's Encrypt) |
| Развёртывание | Docker Compose на арендованном VPS (4 vCPU / 8 ГБ) | Вся система — один compose-файл |

### 8.2 Хостинг: где всё будет работать

Приложение размещается на арендованном VPS — виртуальном сервере у хостинг-провайдера (например, Timeweb Cloud, Selectel, Aeza — в РФ; Hetzner, Contabo — в Европе). Рекомендуемая конфигурация: 4 vCPU, 8 ГБ ОЗУ, 160 ГБ NVMe, гигабитный порт, Ubuntu 24.04 — это ~10–20 $/мес. Домашний компьютер для постоянной работы сервиса не используется и не нужен: VPS работает круглосуточно, не зависит от домашнего интернета и электричества.

### 8.3 Схема компонентов

Все компоненты работают на одном VPS в Docker-сети. Снаружи открыты только: 443 (Caddy → веб-клиент, REST API, WebSocket), 7880-7882 + диапазон UDP (LiveKit, медиатрафик), и SSH по ключу. PostgreSQL, Redis и MinIO наружу не торчат.

Клиент (браузер или Tauri) → Caddy (TLS) → NestJS: REST для CRUD-операций (каналы, роли, история, файлы) и WebSocket для реального времени (новые сообщения, статусы, «печатает», события голоса). Для входа в голосовой канал бэкенд выдаёт клиенту подписанный токен LiveKit, и клиент устанавливает WebRTC-соединение с LiveKit напрямую. NestJS пишет в PostgreSQL, присутствие и события держит в Redis, файлы кладёт в MinIO и отдаёт подписанные ссылки.

### 8.4 Модель данных (основные сущности)

users (id, имя, хэш пароля argon2id, аватар, статус, 2FA-секрет) · invites (код, кто создал, лимит, срок) · categories · channels (тип: text/voice, категория, приватность) · messages (канал, автор, контент, reply_to, attachments[], edited_at, полнотекстовый индекс) · reactions · roles (имя, цвет, permissions-битмаска, позиция) · user_roles · channel_overrides (канал, роль, allow/deny) · dm_conversations + dm_messages · bans, audit_log, pinned_messages, custom_emojis.

### 8.5 Realtime-протокол (WebSocket)

Аутентификация соединения по JWT при рукопожатии. События сервер→клиент: message.new / message.edit / message.delete, reaction.add/remove, typing, presence.update, voice.join/leave/mute, channel.* (изменения структуры), notification. Клиент→сервер: подписка на каналы, typing, ack прочитанного. При переподключении клиент передаёт последний полученный id сообщения и догоняет пропущенное через REST.

### 8.6 Почему этого достаточно для 30 человек

30 пользователей — это десятки сообщений в минуту и максимум ~20 одновременных голосовых потоков. Один процесс Node.js и один инстанс LiveKit держат такую нагрузку с огромным запасом; узкое место — исходящий канал VPS при демонстрациях экрана (примерно 3–5 Мбит/с на зрителя), поэтому VPS выбирается с гигабитным портом. Горизонтальное масштабирование сознательно не проектируем.

## 9. Безопасность

### 9.1 Модель угроз

Реальные угрозы для маленького приватного сервиса: автоматические сканеры и боты, перебор паролей, утечка инвайт-ссылки, XSS через контент сообщений, загрузка вредоносных файлов, захват сервера через незакрытые порты или устаревший софт. Целевые APT-атаки — вне модели.

### 9.2 Меры защиты

- **Транспорт:** только HTTPS/WSS (TLS 1.3, автосертификаты Let's Encrypt через Caddy); HSTS; медиатрафик WebRTC шифруется DTLS-SRTP по умолчанию.
- **Пароли и сессии:** хэширование argon2id; JWT с коротким сроком жизни + ротация refresh-токенов; httpOnly/Secure/SameSite cookie; проверка паролей по базе утечек (haveibeenpwned k-anonymity).
- **Приложение:** валидация всех входных данных (zod/class-validator); санитизация markdown при рендере (никакого сырого HTML) против XSS; строгий Content-Security-Policy; параметризованные запросы через ORM (Prisma) против SQL-инъекций; CSRF-защита; rate limiting на все эндпоинты (вход — 5/мин, сообщения — 10/10 сек, инвайты и загрузки — отдельные лимиты).
- **Файлы:** проверка магических байтов, запрет исполняемых форматов, раздача с отдельного поддомена с Content-Disposition и CSP sandbox, подписанные URL с истечением.
- **Инфраструктура:** SSH только по ключам и не с root; UFW: закрыто всё, кроме 443, портов LiveKit и SSH; fail2ban на SSH и на эндпоинт входа; контейнеры от непривилегированных пользователей; автоматические обновления безопасности ОС; Dependabot/renovate для зависимостей.
- **DDoS:** rate limiting + возможность спрятать 443 за Cloudflare (бесплатный тариф).
- **Резервные копии:** ежедневный дамп PostgreSQL и снапшот MinIO, шифрование, выгрузка во внешнее хранилище (Backblaze B2), хранение 30 дней; ежемесячная проверка восстановления.
- **Мониторинг:** Uptime Kuma (алерт в Telegram при падении), логи через journald/Grafana Loki, алерт при заполнении диска > 80 %.

## 11. Авторские права и юридическая чистота

Создание функционального аналога Discord законно: идеи, концепции и функциональность (каналы, роли, голосовые комнаты) авторским правом не охраняются. Охраняются конкретные выражения: код, графика, тексты, товарные знаки. Правила проекта:

- **Ни строки чужого кода.** Весь код пишется с нуля; не использовать декомпилированный или утёкший код Discord. Открытые библиотеки — можно, с соблюдением их лицензий.
- **Название и знаки.** Не использовать слова «Discord», «Nitro», логотип-геймпад и созвучные названия. Рабочее название «Voxa» перед публичным использованием проверить по базам товарных знаков (Роспатент, EUIPO, WIPO).
- **Визуальный стиль.** Не копировать пиксель-в-пиксель фирменную палитру (#5865F2), логотипы, иконки, маскота, звуки и шрифт ggSans. Наши шрифты и иконки — только с открытой лицензией: Inter (SIL OFL), Lucide (ISC).
- **Лицензии зависимостей.** Основной стек — свободные лицензии MIT/Apache 2.0/BSD/PostgreSQL (React, NestJS, LiveKit, PostgreSQL, Redis 7). Исключение — MinIO (AGPL v3): используется как отдельный сервис через S3-API, код не линкуется с нашим — это допустимо; при желании можно заменить на SeaweedFS или Garage. В CI — автоматическая проверка лицензий (license-checker).
- **Эмодзи.** Использовать открытый набор Twemoji (CC-BY 4.0, с указанием авторства) или Noto Emoji (Apache 2.0), а не эмодзи Discord/Apple.
- **Контент пользователей.** Загружаемые пользователями файлы — их ответственность; в правилах сообщества зафиксировать запрет на распространение пиратского контента.
- **Звуки.** Сигналы входа/выхода/звонка — сгенерировать свои или взять с лицензией CC0.

*Дисклеймер: это не юридическая консультация. Если проект станет публичным open-source или коммерческим — стоит показать юристу.*

## 12. Подробный план реализации

Каждый этап заканчивается работающим, развёрнутым результатом. Этапы выполняются по порядку, без перескакиваний.

### Этап 0. Подготовка

- Завести репозиторий (монорепо: apps/web, apps/desktop, apps/server, packages/shared) с линтером, форматтером, CI (GitHub Actions).
- Арендовать VPS (4 vCPU / 8 ГБ / 160 ГБ NVMe, гигабитный порт), купить домен, настроить DNS.
- Базовая защита сервера: SSH по ключу, UFW, fail2ban, автообновления.
- Каркас docker-compose: PostgreSQL, Redis, MinIO, Caddy, пустой NestJS — проверить, что «Hello world» доступен по HTTPS.

### Этап 1. Фундамент бэкенда

- Схема БД (Prisma) и миграции: users, invites, categories, channels, roles, messages.
- Аутентификация: регистрация по инвайту, вход, JWT + ротация refresh, rate limiting, argon2id.
- CRUD каналов и категорий; система ролей с битмаской прав; middleware проверки прав.
- WebSocket-шлюз: авторизация соединения, комнаты-подписки, событие message.new end-to-end.

### Этап 2. Текстовый чат и веб-клиент

- Каркас React-приложения: роутинг, стор, тёмная тема, трёхколоночный макет.
- Лента сообщений: отправка, получение по WS, markdown, ленивая подгрузка истории, оптимистичный UI.
- Редактирование, удаление, ответы, реакции, упоминания, «печатает», непрочитанные.
- Загрузка файлов в MinIO, инлайн-картинки, предпросмотр ссылок.
- Список участников, статусы присутствия (Redis TTL), профили с аватарами.
- Промежуточный релиз: позвать 3–5 друзей потестировать чат.

### Этап 3. Голос

- Развернуть LiveKit в compose; выдача токенов комнат бэкендом; связать голосовые каналы с комнатами LiveKit.
- Клиент: вход/выход из голосового канала, mute/deafen, индикация говорящих, список участников в канале в реальном времени.
- Выбор аудиоустройств, индивидуальная громкость, звуковые сигналы входа/выхода.
- Демонстрация экрана (720p) со звуком; просмотр выбранного потока.
- Нагрузочный тест: 15 участников в комнате (livekit-cli load-test).

### Этап 4. Десктоп-клиент и полировка

- Обернуть веб-клиент в Tauri: трей, автозапуск, глобальные хоткеи mute/PTT, нативные уведомления, автообновление (tauri-updater).
- Личные сообщения и звонки 1-на-1.
- Модерация: кик, бан, таймаут; журнал аудита; панель владельца.
- Пройтись по UX: пустые состояния, ошибки сети, скелетоны загрузки.

### Этап 5. Безопасность и запуск

- Чек-лист безопасности из раздела 9 целиком; прогнать OWASP ZAP baseline-скан; зависимости — npm audit.
- Настроить бэкапы с выгрузкой в B2 и проверить восстановление на чистой машине.
- Мониторинг: Uptime Kuma + алерты в Telegram; логи.
- Написать правила сообщества; сгенерировать инвайты; онбординг всех 20–30 друзей. Запуск.

### Этап 6. После запуска (по обратной связи)

- v1.1: поиск, пины, кастомные эмодзи, звуковая панель, 2FA, PTT, переопределения прав по каналам.
- v2: «Кинотеатр», камера, групповые ЛС, статистика сообщества, светлая тема, возможная публикация в open-source (лицензия AGPL-3.0 или MIT).
